Over-editing: when AI coding models change more code than necessary
A research blog post making the rounds on Hacker News (336 points) examines a phenomenon called 'over-editing' in AI coding assistants — where a model modifies code beyond what the task actually requires. This is a subtle but real problem: if you ask an AI to fix a bug, and it also reformats your file, renames variables, and refactors a function it decided was suboptimal, you get a diff that's much harder to review and a higher risk of unintended behavior changes. The post argues this is a distinct failure mode from generating incorrect code — the model might produce correct code while still causing harm by changing things that didn't need changing. For teams doing code review, AI-generated PRs that are unnecessarily large increase reviewer fatigue and make it harder to spot actual errors. The issue also connects to trust calibration: if you can't predict what scope of change an AI assistant will make, you'll either over-supervise it (defeating the productivity gain) or under-supervise it (accepting risk). The post proposes some evaluation approaches for measuring minimal editing behavior, which is useful for anyone trying to benchmark coding assistants for production use. This is the kind of nuanced analysis that tends to get lost in the headline benchmark races, but it matters enormously for day-to-day developer experience. It's also a useful framing for product teams building coding tools — 'surgical' editing as an explicit design goal rather than an afterthought. The 336 HN score suggests this resonates with developers who've experienced the problem firsthand.

Firefox and Tor users at risk: stable cross-identity fingerprint found via IndexedDB
Researchers at Fingerprint.com published a significant finding: they discovered a stable Firefox identifier in IndexedDB that links all your private Tor browser identities — meaning even if you're using Tor to maintain separate identities, a tracker exploiting this vulnerability can correlate them. This is a serious privacy vulnerability because Tor's threat model explicitly assumes that different browser sessions should be unlinkable, and IndexedDB is a storage mechanism that should not be persisting identifiable information across private browsing contexts. The vulnerability scored 574 points on Hacker News, reflecting how seriously the security and privacy community is taking it. IndexedDB fingerprinting is a class of vulnerability where browser storage APIs that are supposed to be sandboxed per origin or per session leak information that enables cross-session or cross-origin tracking. For ordinary Firefox users, the practical implication is that private browsing mode is less private than assumed. For Tor users, the implication is more serious: if you're using Tor specifically to maintain separate personas or protect your identity, this is a meaningful threat to your operational security. Mozilla will presumably patch this, but the speed and communication of the fix will matter. It's also a reminder that browser privacy is a complex layered system, and each layer can have unexpected interactions with the others. The Tor Browser team typically ships hardened configurations on top of Firefox, and they'll need to assess whether their current settings mitigate this or not.

A separate Ars Technica story covers NASA's effort to get Apollo-era Moon landing footage in HD using photon counting techniques, which is thematically related: the challenge of getting high-quality data between Earth and space is a persistent thread in space exploration.
