OpenAI designs AI agents to resist prompt injection — here's how
OpenAI has published a detailed post on how ChatGPT's agent workflows are being designed to resist prompt injection and social engineering attacks. The core strategy involves constraining risky actions, protecting sensitive data, and building architectural guardrails into the agent runtime rather than relying solely on model-level instruction following. Prompt injection — where malicious content in the environment tricks an agent into taking unintended actions — is widely considered the most serious near-term security threat in agentic AI systems. The fact that OpenAI is publishing this in detail suggests they've moved from acknowledging the problem to having concrete mitigations worth sharing. The post covers how the Responses API and hosted containers are being used to create isolated, auditable execution environments for agents. One key insight is that security in agentic systems requires thinking about the full action space of an agent, not just its outputs — an agent that can browse the web, execute code, and call APIs has a much larger attack surface than a chatbot. For engineers building on top of OpenAI's agent infrastructure, this is required reading. It also complements the simultaneous work on instruction hierarchy (IH-Challenge), which trains models to prioritize trusted instructions over injected ones. Together these represent a more systematic approach to agent security than has existed before. The honest caveat is that prompt injection remains an unsolved problem at a fundamental level, and these are mitigations, not solutions.

Future AI chips may be built on glass substrates — Absolics starts commercial production
A South Korean company called Absolics is beginning commercial production of glass-based substrates designed for next-generation AI chips, and MIT Technology Review has a detailed piece on why this matters. Traditional chip packaging uses organic materials that have limitations in how densely components can be packed and how efficiently signals travel between them. Glass substrates offer better dimensional stability, higher density interconnects, and lower signal loss — properties that become increasingly important as AI accelerators push toward higher compute density. The timing is interesting: as AI model sizes continue to grow and inference demands scale, the bottleneck increasingly shifts from raw transistor performance to how efficiently data moves between chips and memory. Glass-based packaging addresses some of that. Intel has been working on glass substrates for years but hasn't shipped commercially at scale. Absolics getting to commercial production is a meaningful milestone. The implications for the AI hardware supply chain are potentially significant — this could enable denser, more power-efficient AI accelerators than current organic substrate designs allow. It's a deep infrastructure story that doesn't get the attention it deserves compared to model releases. For engineers thinking about where AI compute is heading over the next five years, substrate technology is one of the less-discussed but genuinely important variables.

Supply chain attack using invisible Unicode hits GitHub and other repositories — 'Glassworm' returns
A sophisticated supply chain attack is targeting GitHub, npm, and VS Code extensions using invisible Unicode characters to hide malicious code from human reviewers. The attack — dubbed 'Glassworm' — exploits Unicode control characters that are invisible in most text editors and code review interfaces, allowing malicious logic to be embedded in code that looks clean on inspection. This technique was known theoretically for years but largely ignored until attackers started operationalizing it. The attack affects the fundamental assumption that code review catches malicious changes — if reviewers literally cannot see the malicious code, the review process provides false assurance. This has serious implications for open source supply chain security, where code review is often the primary defense. The attack targets the trust layer of the development ecosystem: maintainers merging PRs that appear clean, CI/CD pipelines running code that has hidden payloads. Ars Technica and Hacker News both covered this (with the Hacker News entry linking to a detailed Aikido Security breakdown), indicating it's getting significant attention in the security community. Mitigations involve tooling that renders invisible characters visually or flags their presence, and some linters and security scanners are being updated to catch this. For any team that merges external PRs or installs packages from public registries — which is essentially everyone — this is a real and present risk that deserves immediate attention in your security review process.
