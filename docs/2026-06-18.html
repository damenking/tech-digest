RFC 10008 formally defines the HTTP QUERY method — a long-overdue fix for search-over-HTTP
RFC 10008 has been published, officially standardizing the HTTP QUERY method — a new HTTP verb that fills a gap that developers have been working around for decades. The problem it solves: HTTP GET requests can't have a body (or rather, semantics for GET bodies are undefined and unreliable), but many search and query operations need to send complex, structured request bodies rather than encoding everything into URL query parameters. The common workaround has been to use POST for read-only queries, which breaks HTTP caching semantics and creates semantic ambiguity — a POST is supposed to imply a state change, not a read. QUERY gives you a safe, idempotent method with a request body, which means caches can correctly handle it and the semantic contract is clear: this is a read operation with a complex request structure. For API designers, this is a welcome addition that makes it possible to build semantically correct search and filter endpoints without hacking around HTTP constraints. The practical adoption path requires support from proxies, CDNs, load balancers, and HTTP client libraries — which means there'll be a lag before QUERY is safe to use in production without fallback handling. But having the RFC published is the necessary first step. For anyone building search APIs, GraphQL-over-HTTP, or complex filtering interfaces, it's worth reading the spec and tracking library support. This one has been in draft for years and the community is glad it finally landed.

How Browser Use runs Firecracker VMs inside EC2 to start browsers in under 1 second
Browser Use, a startup focused on browser automation infrastructure, has published a detailed technical writeup on how they use Firecracker microVMs inside EC2 instances to achieve browser startup times under one second — a roughly 10x improvement over traditional VM-based approaches. Firecracker is Amazon's open-source VMM (virtual machine monitor) originally built for Lambda and Fargate; it's designed to boot microVMs in milliseconds with minimal overhead compared to full virtualization. The challenge for browser automation is that spinning up a full browser (Chrome or Chromium) involves significant initialization overhead — the browser itself takes time to start, and if you're doing this thousands of times per minute for web scraping, testing, or AI agent tasks, that startup latency compounds into serious throughput limitations. Browser Use's approach snapshots browser state and uses Firecracker's copy-on-write memory to restore clean browser instances from a snapshot rather than cold-starting from scratch. The result is isolation (each session gets a fresh VM), security (cross-session contamination is prevented), and speed (sub-second startup). This matters specifically for the AI agent use case, where agents need to spin up browsers on demand to interact with web interfaces. As AI agents become more prevalent, the infrastructure to run them at scale — fast, isolated, cost-efficient browser sessions — becomes a genuine engineering challenge. The writeup is a good practical example of applying existing cloud infrastructure primitives in non-obvious ways to solve a hard performance problem.
