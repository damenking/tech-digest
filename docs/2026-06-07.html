The incident is a wake-up call that agentic AI systems with real-world consequences need layered authorization controls, not just content moderation. Meta hasn't disclosed the full scope, but the number is confirmed to be in the thousands. For enterprise users, this is a feature worth enabling immediately if you're using ChatGPT with sensitive internal documents. Mark's concern is that as people increasingly outsource thinking tasks to AI — drafting emails, summarizing documents, making decisions — they may be weakening the cognitive muscles that handle those tasks. Running Python in a WASM sandbox with MicroPython — a practical approach to safe code execution
Simon Willison has released micropython-wasm, an alpha library that uses MicroPython compiled to WebAssembly to create a genuinely sandboxed Python execution environment. The goal is code execution safety — running untrusted or AI-generated Python code without giving it access to the host system, network, or file system. Willison has been trying to solve this problem for years and says this approach finally hits the combination of properties he's been looking for: true isolation via WASM's security model, reasonable Python compatibility via MicroPython, and easy integration as a Python package. He's already using it as a code execution plugin for Datasette Agent, and he reports that GPT-5.5 has so far failed to break out of the sandbox — which is a meaningful stress test given how aggressively frontier models probe for escape routes. The CLI addition in the 0.1a2 release makes it easy to use directly without writing integration code, which should accelerate experimentation. For anyone building AI agents that need to execute code — which is increasingly common as agentic workflows mature — sandboxed execution is a critical safety layer. The WASM approach is clever because it reuses a security model that browser vendors have spent years hardening, rather than rolling a custom sandbox. MicroPython doesn't have full stdlib compatibility, so there will be Python code patterns that don't work, but for the code-generation use case that's often acceptable. This is a genuinely useful open source contribution to the AI agent safety toolkit.

Moving beyond fork() + exec() — Linux explores new process spawning primitives
An LWN article exploring proposals to move beyond the decades-old fork()+exec() pattern for spawning processes in Linux has attracted significant attention (288 points on Hacker News), reflecting real pain among systems programmers. The fork()+exec() pattern dates to Unix's earliest days and has well-known problems: fork() copies the entire process address space before exec() throws it away, which is expensive and creates subtle bugs in multi-threaded programs (async-signal-safe code between fork and exec is notoriously hard to get right). Modern alternatives like posix_spawn() exist but have their own limitations. The proposals discussed in the article include more expressive process creation APIs that let you specify the new process's environment, file descriptors, and other properties atomically without the dangerous intermediate forked-but-not-execed state. For most application developers, this is invisible infrastructure — but for anyone writing servers, container runtimes, or build systems that spawn many processes, better primitives could meaningfully reduce complexity and improve reliability. The Linux kernel's conservatism about breaking application compatibility means this kind of change moves slowly, but the discussion is active. Languages like Rust have already been pushing on safer process spawning APIs in their standard libraries, and kernel-level improvements would enable even better solutions. This is the kind of foundational systems work that doesn't make headlines but improves everyone's lives over the following decade.
