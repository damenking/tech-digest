Quantum Error Correction Can Now Continuously Recalibrate a Processor
Researchers have demonstrated a technique where reinforcement learning uses the error information generated during quantum error correction to continuously adjust the control algorithms running on a quantum processor. This is a meaningful step because one of the practical headaches in quantum computing is drift — qubits and their control electronics don't stay perfectly calibrated over time, requiring periodic recalibration that takes the processor offline. By turning the error correction process itself into a feedback signal for continuous recalibration, the system essentially gets smarter and more stable the longer it runs. The reinforcement learning component is what's novel here: rather than relying on a human operator or a fixed recalibration schedule, the system learns to compensate for drift patterns autonomously. This has real implications for the path to practical quantum computing — hardware stability is one of the less glamorous but genuinely hard engineering problems in the field. The technique also points toward a future where quantum processors could maintain coherence for longer operational periods, which is necessary for running the deep circuits that would actually be useful for chemistry simulation, optimization, and cryptography. It's a good paper and worth flagging for anyone tracking quantum hardware maturity, though it's still early-stage research rather than a production-ready system.

GhostLock: A Stack Use-After-Free Bug Has Lived in All Linux Distributions for 15 Years
Security researchers at Nebusec published details of a vulnerability they're calling GhostLock, a stack use-after-free bug that has apparently existed in every major Linux distribution for approximately 15 years. Stack UAF vulnerabilities are particularly dangerous because they can allow an attacker to corrupt stack memory in ways that enable privilege escalation or arbitrary code execution. The fact that this persisted for 15 years across all major distros — Debian, Ubuntu, Red Hat, Arch, and everything else — suggests it was either in very obscure code paths that weren't heavily audited, or in code that was trusted precisely because of its age and apparent stability, a classic Lindy Effect failure mode. The IO subsystem context (based on the nebusec.ai/research/ionstack-part-2 URL) suggests this is related to Linux's IO stack, which handles storage and network operations and is both performance-critical and complex enough that subtle bugs can hide for a long time. Patches are presumably in progress or already available, and this should be treated as a high-priority update for any Linux system. For security engineers, this is a good reminder that age is not a proxy for correctness in low-level systems code, and that use-after-free bugs in the kernel remain one of the most reliably exploitable vulnerability classes.
