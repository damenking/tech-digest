Google releases Gemma 4: Apache 2.0-licensed open models with vision and MoE
Google DeepMind dropped Gemma 4 this week, the first major update to its open model family in about a year. The lineup includes four models: 2B, 4B, 31B, and a 26B Mixture-of-Experts variant (26B-A4B) — all of which are vision-capable and now licensed under Apache 2.0, a significant upgrade from the more restrictive previous Gemma license. The switch to Apache 2.0 is genuinely meaningful: it removes friction for commercial use and integration into downstream products without the headaches of custom license negotiation. Google is leaning hard on the 'intelligence-per-parameter' framing, and early community benchmarks suggest these models punch above their weight class. The MoE variant is particularly interesting — it only activates 4B parameters at a time despite having 26B total, meaning you get larger-model quality at smaller-model inference cost. Simon Willison, who tracks these releases closely, noted this as more evidence that small, efficient models are one of the hottest research areas right now. The vision capability across all sizes is a nice touch — you're not forced to go big just to handle images. For developers who want a capable open model they can actually deploy commercially without legal headaches, Gemma 4 just became a serious contender. The Apache 2.0 move also signals Google is getting more serious about competing with Meta's Llama franchise for developer mindshare. Worth grabbing and benchmarking against your specific use case.

The benchmark problem is compounded by the fact that model providers have incentives to optimize for existing benchmarks (or simply train on benchmark data, intentionally or not) rather than genuine capability improvement. Simon Willison's podcast appearance this week echoed similar themes about the gap between benchmark performance and agentic reliability in production. For engineers evaluating models for real applications, the practical takeaway is: don't trust leaderboard rankings alone — run evals on tasks that actually resemble your use case. New Rowhammer variants (GDDRHammer, GeForge) give attackers full control via Nvidia GPU memory
Security researchers published details on two new Rowhammer attack variants — GDDRHammer and GeForge — that exploit GDDR memory on Nvidia GPUs to ultimately compromise the CPU and gain full machine control. Rowhammer attacks work by repeatedly accessing memory rows to cause bit flips in adjacent rows through electrical interference, and they've been a known threat vector for DRAM for over a decade. Extending this to GDDR GPU memory is a meaningful escalation because GPUs are now central infrastructure in data centers, cloud AI workloads, and high-performance computing clusters. The path from GPU memory to CPU compromise is the particularly dangerous part — it means an attacker who can run code on a GPU (which is increasingly common in multi-tenant cloud environments where GPUs are shared) might be able to escape the GPU context entirely. For cloud providers running GPU clusters for AI training and inference, this is an urgent concern: multi-tenant GPU sharing is how providers maximize expensive Nvidia hardware utilization, and Rowhammer-style attacks that cross that boundary could allow tenant-to-tenant attacks. Nvidia has not yet issued a formal response with mitigations. Memory-level attacks like this are notoriously hard to patch without hardware changes, and GDDR isn't typically designed with the same ECC protections as server DRAM. This one should be on the radar of any security team running GPU infrastructure.
