Canadian election databases used 'canary traps' to catch leaks — and it worked
Canada's election infrastructure quietly employed a classic counterintelligence technique called a 'canary trap' in its voter databases — embedding intentional, unique errors in copies of the data given to different parties or individuals. When the specific error in a leaked copy is identified, it points directly to which copy was compromised, and therefore who leaked it. The technique has a long history in intelligence and publishing circles (publishers sometimes plant unique typos in advance copies to track leaks), but its use in electoral databases is a notable operational security move. The fact that it worked — successfully identifying the source of a database leak — validates the approach for other democratic institutions thinking about how to protect sensitive electoral data. Election database security has become a serious concern in the era of influence operations and foreign interference, and canary traps offer a relatively low-cost way to add accountability without changing how the data is used. The story is a useful reminder that some of the most effective security techniques are social and procedural rather than technical. For security engineers and architects managing sensitive data shared across multiple parties, the canary trap principle is worth understanding and potentially applying. It's also a story about operational tradecraft becoming relevant infrastructure policy, which is an interesting convergence.

Microsoft Edge stores all passwords in plaintext in memory, even when not in use
A researcher disclosed that Microsoft Edge stores all saved passwords in clear text in process memory, even passwords that aren't actively being used in the current session. This is a significant security issue because it means any process with memory read access — including malware, or even legitimate software with insufficient privilege boundaries — can trivially extract every saved password from a running Edge process. Modern password managers are generally expected to keep credentials encrypted in memory and only decrypt them on demand, minimizing the exposure window. Edge's behavior violates this principle by essentially keeping the full credential store decrypted at all times. The disclosure hit Hacker News with nearly 500 points, reflecting widespread concern and frustration from the security community. This is particularly awkward for Microsoft given its significant investment in security marketing and its push for enterprises to trust Edge as a managed browser. The practical mitigation for individuals and enterprises is to avoid using Edge's built-in password manager and instead use a dedicated password manager. Microsoft has not yet issued a formal response, fix timeline, or acknowledgment as of publication. For security-conscious organizations, this is the kind of finding that warrants a rapid internal review of Edge deployment policies.
