Microsoft's Project Solara: an Android OS built for AI agents, not apps
Microsoft announced Project Solara at Build — an Android-based operating system fundamentally redesigned around AI agents rather than traditional apps. The pitch is that instead of launching apps to accomplish tasks, you interact with agents that orchestrate across services, data, and device capabilities on your behalf. This is conceptually related to Microsoft's broader Copilot push, but Solara represents a more radical architectural rethink at the OS level. Microsoft famously missed the mobile app era entirely, and this framing — 'we missed apps, but agents are next' — is both honest and a bit audacious. The practical question is whether users actually want agent-first interfaces, or whether that's a developer fantasy that doesn't survive contact with real people who like tapping app icons. There are also serious privacy and security implications when an OS is designed to have agents acting on your behalf across all your data and services. Android being the base makes sense for hardware compatibility, but it also means Solara lives in Google's ecosystem whether Microsoft likes it or not. This is early-stage stuff — don't expect a consumer device running Solara anytime soon — but it signals where Microsoft thinks computing is headed and where they want to lead rather than follow.

Use your Nvidia GPU's VRAM as Linux swap space
An open source project called nbd-vram has appeared on GitHub, allowing Linux users to use their Nvidia GPU's VRAM as swap space — essentially treating the GPU's fast memory as an overflow for system RAM. This scored 288 points on Hacker News, reflecting genuine interest from developers and ML practitioners who frequently run into memory pressure when working with large models locally. The technique uses network block device (NBD) infrastructure to expose the VRAM as a block device that the kernel's swap subsystem can then use. Performance characteristics will vary significantly: VRAM is much faster than NVMe swap but the PCIe bus adds latency, and GPU memory is also needed by CUDA workloads themselves, so contention is a real concern. That said, for workloads where the GPU is idle (or nearly so) and system RAM is saturated, this could be a practical workaround for running larger models on consumer hardware. It's the kind of clever hack that the open source community excels at — finding creative uses for underutilized resources. Don't expect this to be a smooth production-grade solution yet, but it's worth watching for anyone doing local AI inference who's constantly fighting memory limits.
