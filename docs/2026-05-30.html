Coders refusing to work without AI — and researchers say that could backfire
A growing number of software developers have reached the point where they flatly refuse to work without AI coding assistance, and researchers are starting to ring alarm bells about what that dependency means for long-term code quality. The core concern is that AI tools help developers produce code faster, but the code they produce may be harder to maintain, less well-understood by the humans shipping it, and more prone to subtle bugs that only surface later. There's a compounding problem here: if developers stop deeply reading and reasoning about code because the AI handles the first draft, the skills needed to catch AI errors atrophy. This creates a feedback loop where AI-generated code gets reviewed by developers who are themselves increasingly reliant on AI to understand code — a recipe for accumulating technical debt that nobody fully comprehends. TechCrunch's piece also touches on the job security dimension: developers who've outsourced their foundational skills to AI tools are in a precarious position if those tools change, get expensive, or produce code in a domain where the model has gaps. Cognition's Scott Wu, whose company makes Devin (arguably the most capable AI coding agent), explicitly argues that AI coding agents shouldn't replace humans — which is both a principled stance and probably good marketing. The most useful framing for engineers is probably: AI as pair programmer is great; AI as primary author with human as rubber stamp is risky. The developers who will thrive long-term are likely those using AI to go faster while staying deeply engaged with what the code actually does.

Microsoft Copilot Cowork found to exfiltrate files via prompt injection
Security researchers at PromptArmor discovered that Microsoft Copilot Cowork — Microsoft's AI-powered collaborative workspace tool — could be manipulated into exfiltrating files through a prompt injection attack. The vulnerability follows a now-familiar pattern: an attacker embeds malicious instructions in a document or email that the AI agent reads, the agent then follows those instructions (thinking they're legitimate user requests), and sensitive files get sent to attacker-controlled destinations. This is arguably the defining unsolved security problem for agentic AI systems: agents that have access to files, email, and calendar are inherently high-value targets for prompt injection, and current models aren't reliably able to distinguish between instructions from their actual users and instructions embedded in content they're processing. Microsoft has a mixed track record here — they've been shipping agentic products aggressively while the underlying security model remains immature. What's frustrating is that this isn't a new attack class: researchers have been demonstrating prompt injection against LLM-powered tools for years, and the mitigations remain inadequate. For organizations deploying Copilot Cowork or similar tools with broad file system access, this is a genuine risk that should inform your threat model. The right posture is probably to treat AI agents with file access like you'd treat any third-party application with OAuth access to your data: least-privilege, audit logging, and healthy skepticism about what they're actually doing.

SQLite is all you need for durable workflows
A blog post making the case that SQLite is sufficient infrastructure for building durable, reliable workflows — without reaching for heavyweight distributed systems like Kafka, Temporal, or cloud-managed queuing services — hit the top of Hacker News with nearly 500 upvotes, indicating the argument resonated strongly with the developer community. The core claim is that SQLite's ACID guarantees, combined with WAL mode and careful schema design, give you everything you need for reliable task queuing, workflow state persistence, and exactly-once execution semantics — at a fraction of the operational complexity. This is part of a broader 'SQLite is underrated' movement that's been building over the past few years, accelerated by projects like Litestream, Turso, and the general trend toward single-binary deployments. The counterarguments are real: SQLite doesn't scale across multiple writers or multiple machines the way a proper distributed system does, so there are genuine ceilings on throughput and horizontal scalability. But for the vast majority of applications that don't need millions of events per second, SQLite-backed workflows eliminate an entire category of infrastructure complexity. The post pairs interestingly with the sqlite AGENTS.md file that appeared in the SQLite repository this week — a guide for AI agents working with the SQLite codebase, signaling that even SQLite's maintainers are thinking about how AI interacts with their project. If you're building workflow orchestration for a medium-sized application and reaching for Temporal or a cloud queue by default, this is worth reading first.

For the tech industry, this matters because a significant portion of foundational research in computer science, materials science, and biology flows through federal grants. The proposal is still in comment period, and will likely face legal challenges, but the direction of travel is clear.
