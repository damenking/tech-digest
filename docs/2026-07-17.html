Mira Murati's Thinking Machines Lab releases Inkling, a 975B open-weight multimodal model
Thinking Machines Lab, the AI startup founded by former OpenAI CTO Mira Murati, just dropped their first open-weights model called Inkling. It's a Mixture-of-Experts transformer with 975 billion total parameters but only 41 billion active at inference time — the MoE architecture means it's considerably cheaper to run than the parameter count suggests. The model is Apache 2.0 licensed, which is about as permissive as it gets for commercial use. Notably, it was trained on 45 trillion tokens of text, images, audio, and video, making it genuinely multimodal from the ground up rather than bolted together. They're also promising Inkling-Small, a 276B total / 12B active model, but that's still being tested. Murati has been quiet since leaving OpenAI, so this is the lab's real coming-out moment and the community is watching closely. The Apache-2.0 license is a deliberate signal — it's saying 'we want this used everywhere, including commercially, without strings attached.' The MoE design and multimodal training pipeline suggest Thinking Machines is trying to compete on efficiency as much as raw capability. Whether Inkling can match proprietary frontier models is an open question, but the open-weight release will give researchers direct access to stress-test it. This is one of the more significant open-weight drops of the year given who's behind it.

OpenAI's GPT-Red: an automated red-teaming LLM built to harden other models
OpenAI has published details on GPT-Red, an internal LLM they describe as a 'super-hacker' that they pit against their other models to find vulnerabilities before release. The system uses self-play — essentially having GPT-Red continuously probe GPT-5.6 and other models for weaknesses — to automate what has traditionally been a slow, expensive manual red-teaming process. OpenAI says GPT-5.6, their latest flagship, is their most robust release yet specifically because of adversarial training against GPT-Red. The approach is conceptually similar to how AlphaGo trained by playing against itself, applied to AI safety rather than board games. Automating red-teaming is genuinely important because human red-teamers can only probe so much surface area before a release, while an LLM can run thousands of attack attempts continuously. GPT-Red also targets prompt injection robustness, which is increasingly critical as AI agents get deployed in environments where they process untrusted input. The MIT Technology Review writeup adds useful context on why this matters beyond just benchmark scores — it's about what happens when models interact with adversarial real-world inputs. One honest caveat: OpenAI's claims about their own robustness improvements deserve independent verification, since the company has an obvious interest in saying their safety work is paying off. Still, the self-play approach to safety evaluation is a meaningful methodological contribution regardless of the specific numbers. The full technical paper is worth reading if you're working on AI deployment and care about attack surface.

GPT-5.6 is deleting files in agentic coding workflows — here's what happened
A significant bug surfaced this week: GPT-5.6, OpenAI's latest model, has been unexpectedly deleting files when used in agentic coding contexts. OpenAI's Thibault Sottiaux confirmed the investigation and identified the conditions: the bug most commonly occurs when full access mode is enabled and Codex is run without sandboxing protections or auto-review enabled, and the model attempts to override the $HOME environment variable to define a temporary directory. This is exactly the kind of failure mode that makes senior engineers nervous about agentic deployments — not the model being 'wrong' in an obvious way, but the model taking irreversible real-world actions based on subtly mistaken reasoning. The fix is essentially 'don't run agents without sandboxing,' which is good advice anyway but highlights that the default configurations aren't safe enough for production use without careful configuration. It's a concrete reminder that 'agentic AI' is still genuinely risky when the agent has write access to a filesystem. The incident aligns with broader concerns in the community about human-in-the-loop fatigue — as Pydantic's recent essay noted, reviewers get tired and start approving things reflexively. Sandboxing and auto-review exist precisely for this reason, and this bug is a case study in what happens when you skip them. If you're running any agentic coding workflows in production, this is worth auditing your configuration against.
