Google Shoehorned Rust Into the Pixel 10 Modem to Harden Legacy Code
Google has revealed that the Pixel 10's cellular modem now includes Rust code, a significant engineering achievement given that cellular modem firmware is historically one of the most legacy-entrenched and opaque areas of consumer hardware. Modems are notorious black boxes — they run decades-old C/C++ code, often from third-party vendors, and have been the source of serious security vulnerabilities over the years. The Android security team has been on a multi-year campaign to introduce Rust into safety-critical parts of the Android stack, and extending that to modem code is a meaningful escalation. Memory safety bugs — the class of vulnerabilities Rust is designed to eliminate — are disproportionately exploited in baseband attacks, which can compromise a device at a very deep level, often with no user interaction required. The approach wasn't a clean rewrite; Google literally shoehorned Rust into an existing C codebase, which is technically tricky but increasingly well-supported through interoperability tooling. This follows similar efforts in Linux kernel Rust adoption and Android's broader Rust migration, suggesting an industry-wide momentum shift. It's also a notable data point for the ongoing debate about whether Rust can be practically adopted in embedded and firmware contexts — the answer increasingly appears to be yes, if you're patient and have Google-level resources. For security researchers, it's worth watching how this affects the vulnerability landscape for Pixel 10 baseband attacks compared to previous generations.

LLMs Lack the Virtue of Laziness — And That's a Real Engineering Problem
Systems software veteran Bryan Cantrill wrote a sharp critique of LLMs that's been circulating in engineering circles, and Simon Willison flagged it for good reason. Cantrill's core argument is that LLMs fundamentally lack the programmer's virtue of laziness — the instinct to avoid unnecessary complexity, to do less, to optimize for future maintainability. A lazy programmer writes simple code because they don't want to debug complicated code at 2am six months later. LLMs have no such incentive: generating more code costs them nothing, so they produce layercakes of abstraction and verbosity that technically work but are a nightmare to maintain. This is a subtle but important critique that goes beyond the usual hallucination complaints. Code correctness is tractable — you can run tests. But code quality, simplicity, and long-term maintainability are much harder to measure, and that's precisely where LLM-generated code tends to degrade silently. Cantrill has spent his career at companies like Sun Microsystems and Joyent thinking hard about systems software quality, so his framing carries weight. For teams using AI-assisted coding heavily, this is worth taking seriously — you may be accumulating complexity debt faster than you realize. The fix isn't necessarily to use AI less, but to be more deliberate about reviewing and simplifying AI-generated code rather than accepting it wholesale.

Someone Bought 30 WordPress Plugins and Planted Backdoors in All of Them
In a significant supply chain attack, a malicious actor purchased 30 legitimate WordPress plugins from their original developers and then pushed updates containing backdoors to all of them. This is a textbook supply chain attack: rather than trying to compromise a single high-value target, the attacker acquired trusted, established plugins with existing install bases and poisoned them all at once. The scale is particularly alarming — 30 plugins means potentially hundreds of thousands of WordPress sites were exposed. WordPress's plugin ecosystem is an enormous attack surface: there are tens of thousands of plugins, many maintained by individual developers who may not have strong security practices and who might sell their work without fully understanding the implications. The economics here are troubling — an attacker with modest capital can buy out plugin maintainers and instantly inherit trust and distribution. The backdoors presumably enable remote code execution or data exfiltration on any site running the compromised plugins. For anyone running WordPress-based infrastructure, the immediate action is auditing which plugins you're running and checking their recent update history and ownership changes. This also highlights the broader fragility of package ecosystems — npm, PyPI, and others face similar risks, and the WordPress incident is a timely reminder that "trusted" is a snapshot, not a permanent state.

Stacked PRs — also called stacked diffs or patch stacks — allow you to break a large feature into a series of smaller, dependent pull requests that build on each other, making code review much more manageable. Previously, teams wanting this workflow had to use third-party tools like Graphite, ghstack, or manual rebasing workflows, which added friction and complexity. It aligns GitHub more closely with the workflows Phabricator pioneered at Facebook and that tools like Gerrit support for Android/Chromium development. FBI Takes Down W3LL Phishing Operation That Hit 17,000+ Victims
The FBI has announced the takedown of a large-scale phishing operation centered on the W3LL phishing kit, which was used to compromise over 17,000 victims globally. W3LL is a sophisticated phishing-as-a-service platform that had been operating for several years, offering cybercriminals a polished toolkit for stealing credentials and bypassing multi-factor authentication. The MFA-bypass capability is the technically interesting and alarming part — traditional phishing just stole passwords, but W3LL used adversary-in-the-middle techniques to capture both passwords and OTP/TOTP codes in real time, making even MFA-protected accounts vulnerable. The platform was particularly focused on Microsoft 365 business accounts, making it a significant threat to enterprise environments. W3LL reportedly operated a dark web marketplace where the kit was sold as a subscription service, lowering the technical bar for launching sophisticated phishing campaigns. The FBI's action involved coordinating with international partners, which has become standard for these operations given that the infrastructure and victims are spread globally. For security teams, this is a reminder that MFA is not a silver bullet — hardware security keys (FIDO2) remain the only MFA method that's genuinely resistant to adversary-in-the-middle phishing. Organizations should audit their MFA methods and consider moving away from SMS or TOTP for high-value accounts.

Lean Proof Verified a Program Correct — Then a Developer Found a Real Bug
A developer published a detailed post about a humbling experience: they used Lean, the formal verification proof assistant, to prove a program correct, and then subsequently found an actual bug in it. This sounds paradoxical, but it illustrates a subtle and important point about formal verification — proofs are only as good as the properties you specify. If your proof verifies that the program satisfies property X, but the actual bug manifests in property Y that you didn't think to prove, the verification gives you false confidence. This is sometimes called the "specification gap" — the hard part of formal verification isn't the mechanics of writing proofs, it's correctly and completely capturing what "correct" means for your program. The post is a valuable real-world data point in an era where formal verification tooling is becoming more accessible and AI-assisted proof generation is making it cheaper to write proofs. It doesn't undermine the value of formal verification — proofs are still enormously valuable — but it's a useful corrective against the idea that verified equals bug-free. For engineers exploring Lean or Coq or similar tools, this is required reading: think carefully about your specifications, not just your proofs. The Hacker News discussion around this was lively and substantive, suggesting the community found it genuinely instructive.
